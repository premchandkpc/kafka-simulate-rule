# SDK Specification

## 1. Purpose

SDKs allow applications to embed the FlowRule runtime, submit messages, manage rules, and observe execution. The SDK surface is minimal and stable.

## 2. Core SDK (Go Reference)

### 2.1 Package: `flowrule`
```go
// Engine is the entry point
engine, err := flowrule.New(config flowrule.Config)

// Submit a message
result, err := engine.Submit(ctx, msg flowrule.Message)

// Reload rules
engine.Reload(rules []flowrule.Rule)

// Start/Stop
engine.Start(ctx)
engine.Shutdown(ctx)
```

### 2.2 Types
```go
type Config struct {
    WorkerCount  int
    CreditsLimit int
    Transports   TransportConfig
}

type Message struct {
    ID      uint64
    Body    []byte
    Headers map[string]string
}

type Rule struct {
    ID      string
    Version int64
    Source  string       // DSL source
    Targets TargetMap    // name → URL
}

type Result struct {
    Success bool
    Hops    int
    Errors  []StageError
}
```

### 2.3 Transport Configuration
```go
type TransportConfig struct {
    HTTP   HTTPConfig
    gRPC   gRPCConfig   // optional
}

type HTTPConfig struct {
    Timeout    time.Duration
    MaxConns   int
    TLSConfig  *tls.Config
}
```

## 3. Future SDKs

### 3.1 Rust SDK
- Crate: `flowrule`
- Same API surface as Go
- Uses Rust VM implementation
- WASM plugin authoring support

### 3.2 Java SDK
- Package: `io.flowrule`
- JNI bridge to Rust VM (or pure Java VM)
- Maven/Gradle artifact

### 3.3 Python SDK
- Package: `flowrule`
- C extension calling Go or Rust VM
- pip installable

## 4. SDK Design Principles

- **Minimal surface**: <10 public types, <20 public functions
- **Stable API**: Breaking changes require major version bump
- **Idiomatic**: Follow each language's conventions
- **Zero-cost abstraction**: No unnecessary allocation
- **Context first**: All blocking operations accept context
- **Error types**: Exhaustive matchable error types
