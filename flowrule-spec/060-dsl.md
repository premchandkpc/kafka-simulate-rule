# DSL Specification

## 1. Purpose

The FlowRule DSL is a compact, declarative language for expressing message routing programs. It is designed to be human-writable, machine-parseable, and unambiguous. The DSL is not Turing-complete — it expresses routing topology, not arbitrary computation.

## 2. Design Principles

- **Compact**: One token per operation. No punctuation overhead.
- **Linear**: Programs are sequences of operations. No nesting.
- **Deterministic**: Same input always produces same execution.
- **Verifiable**: All errors are caught at compile time.
- **Transport-agnostic**: No protocol details in the language.

## 3. Grammar

```
program      = statement (whitespace statement)*

statement    = next | parallel | collect | fallback | gate | pipe
             | emit | drop | map | key | split | buffer
             | timeout-mod | retry-mod

next         = "n:" target
parallel     = "p:" target ("," target)+
collect      = "c"
fallback     = "f:" target
gate         = "g:" field operator value
pipe         = "|"
emit         = "e:" target ("," target)*
drop         = "d"
map          = "m:" map-expr
key          = "k:" field
split        = "s:" field
buffer       = "b" digit+

timeout-mod  = "t" digit+      (applies to preceding statement)
retry-mod    = "r" digit+      (applies to preceding statement)

operator     = "==" | "!=" | ">=" | "<=" | ">" | "<" | "contains"
map-expr     = "." field-path  (field extraction)
             | "{" construct   (object construction, future)
field-path   = name ("." name)*
target       = name            (resolved via registry)
name         = letter (letter | digit | "_" | "-")*
digit        = "0" | ... | "9"
letter       = "a" | ... | "z" | "A" | ... | "Z"
```

## 4. Semantics

### 4.1 NEXT (`n:target`)
Deliver the current message to a single target. The call response becomes `LastResponse` for subsequent operations.

```
n:validate
  → Call "validate" with current message body
  → On success: store response as LastResponse, continue
  → On failure: check retry/fallback
```

### 4.2 PARALLEL (`p:target1,target2,...`)
Fan-out the current message to multiple targets concurrently. All targets receive the same input.

```
p:fraud,inventory
  → Call "fraud" and "inventory" concurrently
  → Wait for all to complete (COLLECT)
```

### 4.3 COLLECT (`c`)
Synchronization point after PARALLEL. Waits for all parallel branches to complete. Results are merged as a JSON array.

```
p:a,b c
  → Execute a and b in parallel
  → Wait for both
  → Merged result available
```

### 4.4 FALLBACK (`f:target`)
Conditional target that executes only if the preceding operation failed. Clears the failure state on success.

```
n:primary f:backup
  → Try "primary"
  → If primary fails, try "backup"
  → If backup also fails, message is failed
```

### 4.5 GATE (`g:field==value`)
Content-based conditional branch. Evaluates a field from the current message against an operator and value.

```
g:amount>10000 n:manual-review | n:auto-approve
  → Check if amount > 10000
  → If true: execute next block (n:manual-review)
  → If false: skip to PIPE (|) and execute n:auto-approve
```

### 4.6 PIPE (`|`)
Marks the end of a gate-true block. Used with GATE for if/else semantics.

### 4.7 EMIT (`e:target1,target2`)
Fire-and-forget event emission. Sends the current message to targets without waiting for acknowledgment.

```
n:fulfill e:notify,analytics
  → Call "fulfill", wait for response
  → Fire "notify" and "analytics" asynchronously
```

### 4.8 DROP (`d`)
Terminate message processing. The message is not delivered to any target. Used at the end of filtering pipelines.

```
g:type==spam d
  → If message is spam, drop it
  → Otherwise skip to pipe and continue
```

### 4.9 MAP (`m:.field`)
Extract a single field from the current message body. The extracted value becomes the new message body for subsequent operations.

```
m:.user n:lookup
  → Extract "user" field from message
  → Send extracted value to "lookup"
```

### 4.10 KEY (`k:field`)
Extract a partition key from the message for ordering. Does not transform the message body.

```
k:user_id n:process
  → Extract user_id as partition key
  → Route to "process"
```

### 4.11 SPLIT (`s:field`)
Like KEY but splits the message stream by field value. Future extension for stream partitioning.

### 4.12 BUFFER (`b<count>`)
Buffer messages in memory (count specifies capacity). Future extension for batch processing.

## 5. Modifiers

### 5.1 TIMEOUT (`t<ms>`)
Sets the timeout for the preceding NEXT instruction.

```
t5000 n:slow-service
  → Call "slow-service" with 5-second timeout
```

### 5.2 RETRY (`r<count>`)
Sets the retry count for the preceding NEXT instruction.

```
n:unreliable r3
  → Call "unreliable" with up to 3 retries
  → Total attempts: 4 (1 original + 3 retries)
```

## 6. Examples

```
# Simple validation pipeline
n:validate

# Conditional routing with timeout
t1000 g:amount>10000 n:manual f:fallback

# Parallel fan-out with collect
p:svc-a,svc-b c

# Full pipeline
g:user.tier==premium n:fast-lane g:user.tier==standard n:normal-lane
n:tracking e:analytics,notification
```

## 7. Grammar Invariants

- At least one instruction required
- No orphan modifiers (r/t without preceding instruction)
- PIPE only after GATE
- COLLECT only after PARALLEL
- DROP must be terminal
- No nested PARALLEL
